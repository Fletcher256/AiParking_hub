# 重要代码使用说明

本目录是作品的“重要代码”提交包，面向 SS928 / AI 端侧自主泊车项目整理。代码包保留核心源码、配置、文档、模型附件和下位机工程，便于评审快速定位系统实现。

作品名称：

> 基于 SS928 海鸥派的视觉闭环自主泊车系统

系统目标是以 **SS928 海鸥派** 为端侧主控平台，接入 **OS08A20 摄像头**，运行 YOLO 车位感知和自主泊车控制算法，再通过 **STM32 下位机** 执行底盘动作，实现低速自主泊车闭环。仓库中保留 **GS1860 dToF** 相关 baseline 和调试资料，但 dToF 暂未应用到当前泊车闭环，按未实现/预留内容处理。

## 3 核心代码入口

### 3.1 自主泊车主控制器

```text
source_code/01_pc_tools/board_parking_controller.py
```

板端自主泊车主控制器。它接收 YOLO 或视觉模块输出的车位相对状态，调用规划器选择动作，并向 STM32 发送执行指令。

当前主线控制逻辑：

```text
YOLO 车位多边形 / 车位相对状态
  -> board_parking_controller.py
  -> parking_rollout_optimizer.py
  -> parking_rollout_optimizer_h1.json
  -> STM32 串口执行
```

### 3.2 滚动优化规划器

```text
source_code/01_pc_tools/parking_rollout_optimizer.py
```

规划器从当前车位相对姿态出发，枚举未来若干步候选动作，使用底盘响应模型预测执行结果，对候选动作序列评分，然后只输出最优序列的第一步。车辆停止后重新观察并重新规划。

### 3.3 控制核心公共逻辑

```text
source_code/01_pc_tools/parking_controller_core.py
```

存放无串口副作用、可离线测试的核心逻辑，包括成功判据、感知过滤、配置读取和安全判断辅助函数。

### 3.4 备用线跟随策略

```text
source_code/01_pc_tools/parking_line_follow_decision.py
```

保留历史稳定的 `line_follow` 线跟随泊车策略。正式主线是 `rollout_optimizer`，`line_follow` 作为现场回退方案。

### 3.5 配置文件

```text
source_code/03_configs/parking_rollout_optimizer_h1.json
source_code/03_configs/chassis_kinematics.json
source_code/03_configs/parking_success_criteria.json
source_code/03_configs/perception_filter.json
```

这些文件存放动作模板、底盘运动模型、成功判据、视觉过滤参数等。

### 3.6 dToF 预留资料（未接入当前闭环）

```text
source_code/11_official_SS928_dtof_baseline_src/
```

该目录保留 SS928 中 OS08A20 摄像头和 GS1860 dToF 模块 bring-up 的参考源码。当前自主泊车闭环不依赖 dToF 深度图、UDP 深度数据或深度融合结果；该目录仅作为后续接入 dToF 的 baseline 资料。

### 3.7 STM32 下位机工程

```text
source_code/15_stm32_lower_controller_SS928_hub/
```

STM32 底盘执行板工程，负责电机、舵机、串口协议和底层动作执行。工程文件：

```text
source_code/15_stm32_lower_controller_SS928_hub/LED_1.uvprojx
```

### 3.8 YOLO 模型

```text
source_code/16_yolo_models/
```

当前包含：

```text
best.pt      # 训练阶段保存的 PyTorch 权重
best.onnx    # 面向部署和端侧推理的 ONNX 模型文件
```

模型输出需要经过车位几何解析、置信度过滤和坐标转换，再转换为 `lateral_cm`、`y_dist_cm`、`heading_deg` 等控制器需要的状态量。


### 阶段四：验证 YOLO 车位感知链路

视觉链路目标是把摄像头画面转换成车位相对状态，例如：

```text
lateral_cm
y_dist_cm
heading_deg
vision_confidence
车位线左右边界
```

控制器不直接依赖固定倒车序列，而是依赖这些相对状态进行每步重新规划。

### 阶段五：验证 STM32 底盘执行链路

STM32 工程位于：

```text
source_code/15_stm32_lower_controller_SS928_hub/
```

实车前必须先验证：

```text
STOP 指令有效
状态查询有效
前进/后退方向正确
舵机左右方向正确
短动作执行后能停车
```

### 阶段六：实车低速闭环验证

完整闭环逻辑如下：

```text
摄像头观察车位
  -> YOLO 输出车位状态
  -> rollout_optimizer 规划短动作
  -> STM32 执行动作
  -> 小车停止
  -> 重新观察
  -> 重新规划
```

目标成功判据参考：

```text
y_dist_cm <= 1.5
abs(lateral_cm) <= 2.0
abs(heading_deg) <= 3.0
```

实车测试应在低速、安全、可人工急停的条件下进行。

## 5 硬件与运动安全边界

本代码包包含可能连接真实硬件的脚本。为了避免误触发车辆运动，阅读、检查和复现时请遵守：

- 不要在未确认安全条件前启动 MCU bridge、CAN/串口执行、STM32 bridge、ROS launch 或自动泊车流程。
- 不要在未确认场地和急停条件前运行电机、舵机、刹车、油门或任何底盘动作脚本。
- 不要在未确认硬件连接和供电状态前运行相机采集、预留 dToF 采集脚本或板端自动启动脚本。
- 实车运行必须明确目标主机、动作/策略、最大时长、最大距离、急停方式和安全观察人。
- 如果某个流程要求 `/tmp/parking_armed` 等安全门控文件，缺失时不得继续执行运动相关命令。

演示视频、设计报告和更多补充材料不放入本 GitHub 重要代码提交仓库，按比赛平台附件或后续材料单独提交。

## 7 快速总结

本代码包可以按下面这句话理解：

> `01_pc_tools` 负责感知、规划、部署和测试；`03_configs` 负责泊车参数；`10_board_ss928_files` 负责 SS928 板端 camera 和系统脚本；`11_official_SS928_dtof_baseline_src` 仅保留为 dToF 未实现链路的后续接入参考；`15_stm32_lower_controller_SS928_hub` 负责 STM32 底盘执行。完整系统通过“看车位、算相对状态、规划短动作、STM32 执行、停车再观察”的方式实现低速自主泊车。
